# RAG E2E Evaluation and Website-Native Module Loop - Implementation Plan

> For agentic workers: this plan is for the website stack only. Do not use telegram_bot as part of the loop.

## Goal

Refactor and run the RAG evaluation loop against the website-native stack as the single source of truth for ingestion, summarization, and RAG processing. The loop must improve one `website/features/rag_pipeline` module at a time, run 3-5 iterations on that module before advancing, and verify results on the live production site `https://zettelkasten.in` using Naruto's credentials from `docs/login_details.txt`.

This is a decision because the website stack is the production path users actually exercise, so the evaluation loop must measure the same code path that end users hit.

## Explicit Scope

- In scope: `website/features/rag_pipeline/**`, the website UI, website API, and production Supabase data used by the website stack.
- In scope: end-user verification on `zettelkasten.in`.
- In scope: Playwright CLI first, Chrome DevTools MCP only as a fallback.
- In scope: screenshots saved under `ops/screenshots`.
- In scope: verification against the actual production Supabase project and the Naruto user/account data used there.
- Out of scope: `telegram_bot/**`, Telegram capture flows, Render-specific assumptions, and any Telegram-only ingestion or summarization path.

## Operating Rules

- Treat the website stack as the source of truth for ingestion, summarization, and RAG processing.
- Improve each website-native RAG module for 3-5 iterations before moving to the next module.
- Keep the loop module-local; do not mix Telegram work into the same iteration.
- Use the live site for end-user quality checks, not a local mock or a staging clone.
- Use the real production Supabase project for verification of RAG artifacts tied to Naruto.
- Prefer Playwright CLI for browser verification. If the CLI cannot complete a required browser task, fall back to Chrome DevTools MCP.
- Save browser screenshots under `ops/screenshots` with iteration-prefixed filenames.

## Phase 0 - Reconcile the Existing Plan

The old plan assumed a Telegram-led loop and included Render-oriented deployment steps. Remove those assumptions and keep only the website-native flow.

Checklist:

- [ ] Remove any Telegram-bot loop steps.
- [ ] Remove any Render-specific deploy or verification assumptions.
- [ ] Keep only website-native module work.
- [ ] Keep only live production verification on `zettelkasten.in`.
- [ ] Keep only production Supabase verification for the Naruto account and the relevant RAG pipeline artifacts.

## Phase 1 - Define the Website Module Loop

Work through the current `website/features/rag_pipeline` modules in a fixed order, one module at a time. Each module gets 3-5 iterations before advancing unless the data clearly shows no further gain.

Recommended module order:

1. `ingest`
2. `retrieval` and query handling
3. `rerank`
4. `context`
5. `generation` and critic

For each module:

- [ ] Identify the exact website-native files and functions that belong to the module.
- [ ] Pick the narrowest improvement that should move the module score up.
- [ ] Implement the change only in the website stack.
- [ ] Run targeted tests for the touched module.
- [ ] Deploy or refresh the live website path used for verification.
- [ ] Verify the end-user flow on `zettelkasten.in`.
- [ ] Check the production Supabase artifacts for Naruto.
- [ ] Record the iteration result before deciding whether to stay on the module or advance.

Advance to the next module only after 3-5 iterations on the current module or after the module clearly plateaus.

## Phase 2 - End-User Production Verification

Verification must be done as an end user on `https://zettelkasten.in` with Naruto's login details from `docs/login_details.txt`.

Checklist:

- [ ] Open the live site and log in as Naruto.
- [ ] Exercise the website-native RAG flow end to end.
- [ ] Confirm the visible output is usable by an actual end user.
- [ ] Capture screenshots for the verification run under `ops/screenshots`.
- [ ] Use Playwright CLI first for all browser actions that it can handle.
- [ ] If Playwright CLI cannot complete a required browser task, switch to Chrome DevTools MCP for that task only.

Screenshot rules:

- Use iteration-prefixed names such as `iter03-home.png`, `iter03-capture.png`, and `iter03-rag-result.png`.
- Keep screenshots in `ops/screenshots`.
- Capture only what is needed to prove the live user flow worked.

## Phase 3 - Production Supabase Verification

Verify against the actual production Supabase project used by the website stack, not a local database and not a staging clone.

Checklist:

- [ ] Confirm Naruto's capture touched the expected production RAG tables or artifacts.
- [ ] Inspect the relevant `rag_pipeline` tables, views, or generated artifacts for the current module.
- [ ] Confirm the data written by the website matches the live user action.
- [ ] Record any missing or inconsistent rows before advancing the loop.

If the module is supposed to write derived RAG artifacts, verify those artifacts in production Supabase directly for Naruto.

## Phase 4 - Evaluation Loop Discipline

Keep the loop simple and repeatable:

1. Make one website-native module change.
2. Run the module tests.
3. Verify the live user flow on `zettelkasten.in` with Naruto.
4. Verify the corresponding production Supabase artifacts.
5. Score the iteration.
6. Stay on the same module for up to 3-5 iterations.
7. Move to the next module only after the module is improved or plateaus.

Stop conditions:

- The current module has had 3-5 iterations and no longer shows meaningful improvement.
- The overall evaluation reaches the agreed target.
- The iteration cap for the loop is reached.
- Live verification or Supabase verification exposes a blocking regression.

## Phase 5 - Remove Invalid Assumptions From the Old Plan

The following assumptions from the previous plan must not survive the rewrite:

- Telegram-bot-led ingestion or summarization.
- Render-specific deployment or verification language.
- Any local-only proxy for production Supabase verification.
- Any browser verification that treats Playwright MCP as the first choice.
- Any plan step that does not verify the actual end-user flow on `zettelkasten.in`.

## Completion Criteria

This plan is ready when:

- The scope is website-only.
- The per-module loop is explicit and limited to 3-5 iterations per module.
- Production verification is clearly tied to Naruto on `zettelkasten.in`.
- Browser verification prefers Playwright CLI and saves screenshots under `ops/screenshots`.
- Chrome DevTools MCP is documented as the fallback only.
- Production Supabase verification points at the real production project and the relevant `rag_pipeline` artifacts.
- No Render or Telegram assumptions remain.
