# AGENTS.md

This file provides guidance to **Codex CLI** when working in this repository.

> **Canonical instructions live in `CLAUDE.md`.** Codex must read `CLAUDE.md` at session start and treat it as the authoritative project guide (architecture, commands, deployment, configuration, testing, UI constraints, git conventions). This file only holds **Codex-specific additions** — anything not covered here defers to `CLAUDE.md`.

## Why this split

Both Claude Code and Codex CLI edit this repo against the same git checkout. Keeping a single source of truth (`CLAUDE.md`) and a thin Codex pointer (`AGENTS.md`) prevents the two files from drifting — a problem we hit previously when `AGENTS.md` described Render/3-model-cascade while `CLAUDE.md` had already moved to DigitalOcean/10-key-pool.

## Non-negotiable rules (mirrored from CLAUDE.md)

- **No `Co-Authored-By` lines** in commits. Ever. Short, precise, conventional-style prefixes only (`feat:`, `fix:`, `refactor:`, `docs:`, `test:`).
- **No purple in UI.** Knowledge Graph is amber/gold (`#D4A024`); main site is teal.
- **Secrets get `<private>` tags.** Wrap every reference to `GEMINI_API_KEY(S)`, `TELEGRAM_BOT_TOKEN`, `WEBHOOK_SECRET`, `SUPABASE_SERVICE_ROLE_KEY`, `GITHUB_TOKEN`, droplet SSH material, and any file content from `new_envs.txt`, `.env`, or `supabase/.env` before emitting it. Only `ops/.env.example` (template) is exempt.
- **Never invent env-var tables in README.** Keep project-structure sections to major folders only.
- **`get_settings()` in tests must be `@patch`-mocked** — calling it unmocked triggers `SystemExit(1)`.

## Code navigation

Codex doesn't have claude-mem's `smart-explore` skill. For code files, use Codex's native tools (`Read`, `Grep`, `Glob`) with the same *intent* that smart-explore encodes:

1. **Before reading a full file**, check whether a grep for the target symbol + reading a narrow line range can answer the question. Only read end-to-end when you need ≥70% of the file.
2. **Before globbing broadly**, check whether the folder-level `CLAUDE.md` (auto-generated by claude-mem — see below) already tells you where the relevant symbol lives.
3. **For cross-cutting searches**, grep first, then read only the hits — don't read adjacent files "for context" unless a hit is ambiguous.

## Shared memory: claude-mem MCP (read-only for Codex)

This repo uses the **claude-mem** plugin (installed under Claude Code) to capture an observation timeline across sessions. Codex can read the same memory database via the claude-mem MCP server.

**Setup** (already wired in `~/.codex/config.toml` under `[mcp_servers.claude_mem]`):

- MCP server: `C:\Users\LENOVO\.claude\plugins\cache\thedotmack\claude-mem\10.5.6\scripts\mcp-server.cjs`
- Data: `C:\Users\LENOVO\.claude-mem\claude-mem.db` (~30 MB SQLite, shared with Claude Code, no duplication)

**Tools exposed to Codex:**
- `search(query, project="Obsidian_Vault", obs_type=?, dateStart=?, limit=20)` — get the observation index (~50-100 tokens per hit)
- `timeline(anchor=ID, depth_before=3, depth_after=3)` — chronological context around an ID
- `get_observations(ids=[...])` — fetch full narratives for specific IDs (~500-1000 tokens each)
- `smart_search`, `smart_outline`, `smart_unfold` — AST-based code exploration (same 3-layer workflow as in Claude Code)

**3-layer recall workflow (same as Claude Code):**
1. `search` to get an ID index — never fetch full details first.
2. `timeline` around interesting IDs for chronological context.
3. `get_observations(ids=[...])` to batch-fetch the ones you actually need.

**Writes are asymmetric** — Codex cannot capture observations automatically (no `PostToolUse` hooks in Codex). Claude Code is the only writer. If a Codex session produces significant work, at the start of the next Claude Code session ask: *"Summarize what changed in the repo since commit &lt;sha&gt; and record the significant decisions as observations."* Claude's hook flow will generate the observations.

## Folder-level CLAUDE.md timelines

claude-mem auto-generates per-folder `CLAUDE.md` files with recent activity timelines (flag: `CLAUDE_MEM_FOLDER_CLAUDEMD_ENABLED=true`). Target folders: `telegram_bot/pipeline/`, `website/features/`, `supabase/website/kg_public/`, `tests/`.

Codex reads these as normal Markdown files — no MCP tool needed. When you `cd` or `Read` into one of those folders, check for a `CLAUDE.md` sibling and load it first to see what recently changed there.

## Memory tagging (optional but recommended)

When Codex makes a significant choice — particularly any **architectural or design decision** (GeminiKeyPool routing changes, Supabase vs file-store precedence, blue/green cutover, new source extractor, orchestrator sequence changes, auth/secret model changes, dependency/Python-version bumps, test strategy shifts) — open the response with `"This is a decision because..."` followed by the rationale. The Claude-Code-driven handoff summary will pick this up and record it as a `decision`-typed observation in claude-mem.
